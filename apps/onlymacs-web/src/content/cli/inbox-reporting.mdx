# Inbox And Reporting

OnlyMacs operates on a principle of deliberate action: it generates solutions, but you decide when and how they enter your project. This separation is enforced by two core systems. First, the **inbox** acts as a staging area where every patch, script, or configuration file waits for your inspection before touching your codebase. Second, **reporting** handles anonymized telemetry from public swarm runs, allowing the system to improve reliability across the network without exposing your proprietary logic or file paths.

Understanding these systems prevents accidental overwrites and ensures you maintain full control over your development environment. The inbox is your safety net. Reporting helps improve system stability. This page explains how to review artifacts safely, apply changes with confidence, and manage telemetry settings to fit your workflow.

<Learn>
The inbox is a review gate for generated work. No file generated by OnlyMacs modifies your project directory until you explicitly run an apply command. Reporting is opt-in by design for sensitive environments, though enabled by default to help tune model performance. You can inspect, modify, or discard any inbox item, and you can augment automated reports with manual context at any time.
</Learn>

## Reviewing and Applying Inbox Results

When OnlyMacs completes a task, it writes the output to a local inbox directory rather than your source tree. This design mimics a code review process: the agent proposes a change, and you act as the reviewer. Whether the output is a complex refactoring patch, a new configuration file, or a security audit summary, it remains isolated until you verify its intent and correctness.

You should use the inbox workflow for every automated task that produces artifacts. This includes fixing linting errors, generating boilerplate, or updating dependencies. Even if you trust the agent's history, edge cases in your specific project structure can lead to unexpected results. Treating every inbox item as untrusted code until proven otherwise is the safest default behavior.

To begin, inspect the metadata of the most recent item. This gives you a high-level view of what changed, the risk level, and the triggering command.

```bash
onlymacs inbox latest
```

The output provides context such as the artifact type (e.g., `patch`, `script`, `report`), the confidence score assigned by the agent, and a summary of the impact. If the metadata looks reasonable, proceed to examine the actual content. For text-based files like code patches or JSON configs, you might view them directly in the terminal, but for binary files or rich documents, use the open command to launch them in your default system viewer.

```bash
onlymacs open latest
```

This command respects your operating system's file associations, opening code in your editor and PDFs in your previewer. This step is important for verifying that a shell script doesn't contain destructive commands or that a config update doesn't overwrite custom settings. Once you are satisfied, apply the change to your project.

```bash
onlymacs apply latest
```

This moves the artifact from the inbox into your working directory, preserving permissions and adhering to any team policies configured in your environment. If the application fails due to conflicts or policy restrictions, the command will halt and explain the issue rather than forcing the change.

<BestDefault>
Always run `onlymacs inbox latest` followed by `onlymacs open latest` before applying. Treat the inbox as a mandatory review step, even for small fixes. This habit catches subtle errors like off-by-one line numbers in patches or incorrect environment variable names that metadata summaries might miss.
</BestDefault>

<Avoid>
Avoid automating the `apply` step in scripts or CI pipelines without human verification. The inbox exists specifically to break the chain of blind automation. Also, avoid assuming `latest` always refers to the same item across different terminal sessions; the inbox rotates based on recency, so verify the timestamp or ID if there is any delay between generation and review.
</Avoid>

<Recipe>
**Workflow: Safely Applying a Lint Fix**

1.  OnlyMacs finishes a task and notifies you of a pending patch.
2.  Check the summary to ensure it targets the correct file and scope:
    ```bash
    onlymacs inbox latest
    ```
3.  Open the patch in your editor to verify the specific lines changed:
    ```bash
    onlymacs open latest
    ```
4.  Confirm the changes remove unused imports without altering logic, then apply:
    ```bash
    onlymacs apply latest
    ```
5.  Run your local test suite to ensure the applied patch didn't break anything.
</Recipe>

## Understanding Public Run Reporting

Public swarm runs allow OnlyMacs instances to collaborate on complex tasks by distributing work across multiple agents. To maintain reliability and improve success rates, the system collects minimal telemetry from these runs. This data helps identify failing commands, model-specific issues, and operational bottlenecks across the user base.

Importantly, this telemetry is strictly anonymized. Your actual source code, prompt text, local file paths, and generated artifacts stay on your machine unless you explicitly share them. The system only transmits metadata about the process itself, such as which model was used, how long the task took, and whether it succeeded or failed.

<Decision>
**What Data Is Shared?**

| Data Type | Included in Report? | Reason |
| :--- | :--- | :--- |
| Command invoked (e.g., `fix`, `generate`) | Yes | Identifies high-failure workflows |
| Outcome status (success, failure, timeout) | Yes | Tracks overall reliability trends |
| Provider and model identifiers | Yes | Detects model-specific regressions |
| Event counts (retries, fallbacks) | Yes | Measures system load and latency |
| High-level summary (e.g., "resolved 3/5 issues") | Yes | Provides qualitative context |
| Full prompt text | **No** | Protects intellectual property |
| Generated code or artifacts | **No** | Prevents data leakage |
| Local absolute file paths | **No** | Keeps internal structure private |

</Decision>

You should pay attention to reporting behavior if you work in a regulated industry like finance or healthcare, where even metadata transmission might require approval. Similarly, if you are debugging why a swarm run performed poorly, understanding what telemetry is sent can help you correlate local logs with system-wide incidents. For most developers, leaving reporting enabled is the best choice because it improves the tool for everyone with negligible overhead.

## Managing Reporting Preferences

You have full control over telemetry. You can check the current status, disable it for compliance reasons, or re-enable it later. These settings only affect future runs; they do not retroactively delete or alter data already submitted.

To verify your current configuration, run:

```bash
onlymacs report status
```

The output will clearly state whether reporting is enabled or disabled and show the timestamp of the last submission. If you need to stop telemetry immediately, perhaps on a shared CI runner or a secure workstation, use the disable command.

```bash
onlymacs report disable
```

To resume contributing to system improvements later, simply run:

```bash
onlymacs report enable
```

Disabling reporting does not degrade the performance of your local swarm runs or limit feature availability. It only stops the outbound flow of anonymized metrics.

### Adding Manual Context

Automated telemetry captures the "what" and "when," but sometimes it misses the "why." If a run failed due to a transient network blip rather than a logic error, or if the output was perfect despite a timeout warning, you can attach a manual note to the report. This helps engineers tune thresholds and distinguish between true failures and environmental noise.

Use the `--report` flag to append a specific message to the latest run's telemetry.

```bash
onlymacs report latest --report "Quality was good; one provider retry added about 40 seconds due to API lag."
```

This command sends your message alongside the standard metrics immediately. Be specific in your notes: mention durations, observed quality, or external factors. Vague comments like "worked fine" provide little value for debugging systemic issues.

<Example>
**Scenario:** A swarm run timed out after 60 seconds, but the agent actually produced correct code just 0.8 seconds over the limit. The automated report marks this as a failure.
**Action:** Add context to clarify the success.
```bash
onlymacs report latest --report "Correct output generated; failure due to hard time limit exceeded by 0.8s"
```
**Result:** Engineers see that the logic worked and may adjust timeout thresholds for similar tasks, improving future success rates without you ever sharing the code.
</Example>

## Summary: Safe, Controlled Automation

OnlyMacs separates the generation of solutions from their application. The inbox enforces a deliberate review process, ensuring you do not lose control of your codebase to automation. Reporting provides a feedback loop for system improvement while rigorously protecting your privacy through anonymization.

By consistently inspecting inbox items before applying them and managing reporting settings to match your security requirements, you gain the speed of AI assistance without sacrificing safety or compliance. Treat every artifact as a proposal, verify it thoroughly, and contribute context when the automated metrics don't tell the whole story.
